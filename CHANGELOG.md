- Fix a latent wrong-voice bug in RVC model caching, found during the same-day pipeline-speed investigation. The AICoverGen cog behind our pinned RVC version caches the downloaded custom model in a folder named after the download URL's last path segment (`splitext(url.split('/')[-1])`) and SKIPS the download when that folder already exists on the instance. Our model proxy URL ended in the constant `model.zip`, so every voice shared the single cache key "model" — two different voices converted back-to-back on the same warm (shared-pool) instance would make the second job silently convert with the FIRST voice's weights. Fix: voice-convert now builds the proxy URL as `/api/voice-model/<voiceId>/<voiceId>-<sha1(model_path)[:8]>.zip` — unique per voice, and the model_path hash also busts the cache after a retrain that persists a new path (relevant to the planned Raju dry-mic retrain). The proxy route itself ignores the filename segment (it signs from the DB's model_path), so only its doc comment changed; the new name is ~49 chars, far under the 255-char limit the proxy exists to dodge. Side benefit: warm-instance model caching now works CORRECTLY (legitimate skip of the ~120 MB re-download when the same voice runs again). §9 gotcha updated: never revert to a constant filename.
