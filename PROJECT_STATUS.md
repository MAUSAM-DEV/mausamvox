## 2. Architecture Overview

Next.js App Router monolith. `src/middleware.ts` refreshes Supabase sessions on every request. Three Supabase clients in `src/lib/supabase/`: `client` (browser), `server` (RSC/route, anon), `admin` (service-role, server-only, bypasses RLS).

**Surfaces (pages):**
- `/` — landing (`src/components/landing/*`)
- `/auth/*` — sign-in, sign-up, forgot/update password
- `/onboarding` — persona-driven first-run journey
- `/dashboard` — hub
- `/voice-swap` — upload track → split → swap vocals → preview → result
- `/voice-lab` — guided voice cloning wizard (express + studio/pro record)

**API routes** (`src/app/api/`): auth, credits, voice-convert (RVC), voice-lab (clone CRUD + train + sample/presign), voice-swaps (persist/delete + signed proxy), stem-split / karaoke-split / gender-split, prepare-dataset, upload-stem (+presign/sign), stems/download, stems/refresh (batch re-sign of durable stem paths — used on cache restore). Proxy routes (`voice-model/[voiceId]/[filename]`, `voice-swaps/[swapId]/[filename]`) sign storage objects on-read to avoid stale-URL failures. Shared stem copy-to-Supabase helper lives in `src/lib/stem-persist.ts`.

The word clock is a `requestAnimationFrame` loop reading `audioRef.currentTime` (NOT the ~4/s `timeupdate` event); it toggles classes imperatively on spans queried from the current line's DOM node (robust to seek direction), touches the DOM only when the active word changes, and runs **no per-frame React state**. `displayItems` is memoized so `words` keep a stable identity across time-ticks — that's what lets `WordLine`'s `memo` skip re-renders and preserve the imperatively-set classes. Fallback is per-LINE: a line renders word spans only when it has a `words` array AND the host wired an audio element; old rows (no words) and edited lines (edit save now strips words locally too) keep whole-line highlighting. rAF is cancelled on pause / line change / lyrics change / unmount. Both hosts pass `audioRef` + `playing` (PerformanceMode: its audio + playing; KaraokePanel: backing element + `recState==='recording'`). Gaps + the item-3 countdown untouched. Renderer only — the generation route was NOT touched.
- ✅ **Hum-hallucination filter BUILT (2026-07-06, commit `4be0b03`) — ⚠️ untested on live.** Whisper invents repeated gibberish over non-verbal vocal stretches (humming/yodelling/held notes). **Confirmed the engine exposes NO confidence signals** — `no_speech_prob` / `avg_logprob` / `compression_ratio` are all absent (the insanely-fast-whisper cog uses the HF Transformers ASR pipeline → `{ text, chunks:[{text,timestamp}] }` only; those fields come from openai-whisper / faster-whisper). So we can't threshold on confidence without switching engines (rejected — would regress all-language word timestamps). Instead `dropHallucinatedWords(words)` (route.ts) runs at the word level before grouping (after per-word romanization for the romanized path) and strips ONLY the clear machine signature: a maximal run of one normalized token (case- + Latin-diacritic-insensitive, punctuation-stripped; Devanagari matras preserved) or a repeated A-B bigram that is BOTH long AND crammed denser than singable. **Thresholds (tunables at top of the fn): unigram ≥6 identical / bigram ≥4 cycles / AND ≥6 tokens-sec density (span=0 → infinite).** Removed words fall through to the existing gap/♪ + countdown path (no empty line objects; all-filtered → honest no-lyrics path, nothing stored/charged). **Can fix:** repetitive gibberish over hums. **Cannot fix:** a wrong-but-fluent guess (no signal); may rarely drop a genuinely repeated sung hook (conservative thresholds + edit modal are the mitigation). A passive one-line log of the prediction output's top-level + first-chunk keys runs each time, so if the engine ever exposes confidence fields we learn for free. Existing garbage rows need a 25-cr regenerate; no migration/grant/column. Caveat copy updated in LyricsPane.
- 🔍 **INVESTIGATION (2026-07-05) — word-by-word lyric sync (`timestamp: 'word'`). Findings below; BOTH steps now built (above) — kept for the design rationale.**
  - **Output shape:** `vaibhavs10/incredibly-fast-whisper` with `timestamp: 'word'` returns the **same** `{ text, chunks: [{ text, timestamp: [start, end] }] }` shape we already parse — just one WORD per chunk instead of one phrase. `parseChunks` already handles it structurally (null-end guard etc.). Word timings from this cog are heuristic (no forced-alignment model), so they can drift a little within a line — an honest caveat, same spirit as the existing ones.
  - **One call or two?** A single call returns ONE granularity (word OR chunk, not both). BUT a word-level call is a **superset**: we group the words into display lines ourselves (by punctuation / a gap threshold / max-words), so **one word-level transcription serves both the line display AND per-word highlighting — no double prediction, no double charge.** Trade-off: our grouping is slightly cruder than Whisper's native 'chunk' phrase segmentation; acceptable, and the edit modal already lets users fix line splits' text.
  - **Existing rows:** stored with 'chunk' → phrase timings only, no word data. They'd need a **regenerate (25 cr)** to gain word highlighting. New generations would switch to 'word' and get it for free. (Regenerate flow already exists.)
  - **Storage:** nest an optional `words` array inside each existing line object: `{ start, end, text, words?: [{ start, end, text }] }`. `lines` is already schema-less jsonb — **no migration, no new column, no new grant.** Old rows simply lack `words` and render exactly as today (line-level highlight). `text` stays the whole line, so the edit modal and any non-word-aware read still work. **Edit caveat:** editing a line's text invalidates its word split → on edit we'd drop that line's `words` and it reverts to whole-line highlight (documented behaviour, not a bug).
  - **Romanization interaction:** the Hindi-romanized path must transliterate **per word** (romanize each word's Devanagari, line text = the joined words) so word spans keep their timings. The `romanize.ts` fixes are already whole-word, so they carry over cleanly. This is the fiddliest part but tractable.
  - **Renderer:** only the CURRENT line renders as per-word spans (done/upcoming lines stay whole-text and cheap); a word is "sung" once `time ≥ word.start`. To highlight smoothly, drive the clock with `requestAnimationFrame` reading `audio.currentTime` instead of the `<audio>` `timeupdate` event (which fires only ~4×/s — too coarse for words).
  - **Phone performance:** the rAF loop re-rendering one line of ~5–15 spans at up to 60fps is the only new cost. Fine on modern phones, but worth minimizing React churn (compute the active-word index and toggle a class, rather than rebuilding spans each frame). Memory unchanged. Performance Mode already holds a wake lock during playback.
  - **Effort:** Medium — ~a focused session: API (switch to 'word', group→lines, per-word romanize, store nested `words`), renderer (rAF clock + current-line word spans + active-word logic), edit-modal drop-words handling, and one caveat string. No DB/grant/dependency changes.
  - **Recommendation:** viable and cheap infra-wise; the real costs are (a) users must pay 25 cr to regenerate existing tracks and (b) the per-word romanization + rAF renderer are the careful bits. **Stop here for founder approval before building.**

## 9. Known gotchas (reference, do not re-debug)

These are settled. They cost real time to find — don't rediscover them.

- **MVSEP male/female stem labels can be unreliable** for some tracks — the "male"/"female" split occasionally mislabels or bleeds. Treat gender-split output as a hint, not ground truth.
- **Replicate filename-length bug** — Supabase signed URLs end in `?token=<300+ char JWT>`; the RVC container derives its local filename from the URL's last path segment without stripping the query string, hitting the OS 255-char limit (Errno 36). **Solved** via the `/api/voice-model/<id>/…` proxy route that 307-redirects to the real signed URL. ⚠️ **That last URL segment is also the container's MODEL CACHE KEY** (it extracts to a folder named after it and skips the download if the folder exists on a warm instance). The original constant `model.zip` made every voice share one cache entry — wrong-voice weights could be silently reused across back-to-back swaps. Fixed `184b06f`: voice-convert now builds `<voiceId>-<sha1(model_path)[:8]>.zip`, unique per voice and per retrain. Never revert to a constant filename.
- **Use `@breezystack/lamejs`, not `lamejs`** — the original `lamejs` package is broken/unmaintained for our MP3 encoding path.
- **Use `undici` with `allowH2: false`** for Supabase calls from Node — forcing HTTP/1.1 avoids HTTP/2 issues against Supabase.
- **Node v26 HTTP/2 flood-protection issue** — Node 26's HTTP/2 stack trips flood protection on certain request patterns; another reason the `undici` / `allowH2: false` path above is required.
- **Supabase Free 50 MB object cap** — trained RVC model zips are 116 MB+, which exceeds the Free-tier storage object limit. **Required upgrading to Supabase Pro** to store models.
- **Stem-split is `cjwbw/demucs` running 4-stem `htdemucs`** — the cog's stem-selection param is **`model_name`** (not `model`); pass it explicitly. Its output object **always has 6 fixed keys** (`bass, drums, guitar, other, piano, vocals`), but under `htdemucs` **`guitar` and `piano` come back empty** — so 6 keys in the logs does NOT mean 6-stem compute. To actually get guitar/piano you'd set `model_name: 'htdemucs_6s'` (slower, ~1.5–2×) and stop discarding them in `extractStems`. (settled `401a5fe`; don't re-debug) **Marketing copy is now aligned to 4-stem** (landing Features/TechSection/Pricing-Starter; `76a0164`) — if 6-stem ever ships, those strings need to flip back.
